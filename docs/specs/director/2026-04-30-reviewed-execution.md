# Reviewed execution: planned, briefed, reviewed phases

## Summary

The Director takes ownership of the loop's intent. At the start of a run, it produces a canonical typed plan: a graph of phases with explicit acceptance criteria and dependencies. Per phase, it briefs the Executor with scope, files of interest, and out-of-scope guards. After each phase, it reviews the result against the criteria it itself authored, and either approves, sends back with concrete feedback, or escalates to the user. This replaces the current "go do the next item, give it your best" model with steered execution. It is the heart of the Director hypothesis.

## Problem

### Context

bcc today runs the Executor against the whole spec, on a loop. The Executor picks the next item from the plan, does it, writes a journal entry, exits. bcc reads the journal, decides continue/stop, repeats. This works for short specs. On long specs, the Executor's context fills up across iterations within a phase and across phases as the journal grows. Drift, premature `ok` declarations, and silent scope changes become measurable.

The current safeguards (Plan checkboxes, journal Result enum, HEAD-advanced check) are mechanical: they tell us *whether* the Executor said it did something. They do not tell us *whether what it did meets the bar* the spec set.

### User pain

1. **Premature `ok`.** The Executor declares phase 3 done. The journal entry is well-formed. The diff is missing the test the spec asked for. The user only finds out hours later.
1. **Silent scope drift.** The Executor implements phase 3 plus some adjacent cleanup that was not in the spec. The cleanup breaks something else. The journal does not flag it because, by the Executor's own reasoning, it is "in the spirit of the spec".
1. **No mid-run pivot.** The Executor's plan is whatever it inferred at start. Halfway through, it could be obvious that phases should have run in a different order, but neither the Executor nor bcc knows.
1. **Re-discovery cost per iteration.** The Executor re-reads the spec each iteration to figure out what to do next. That is wasteful and noisy; tokens go to navigation rather than work.
1. **Generic prompts.** The current per-iteration message is essentially "do the next item". An Executor with proper scope, criteria, and out-of-scope guards is a different (better) Executor.

### Impact of not solving

bcc stays usable for short, contained specs. Long specs continue to require human supervision proportional to their length, eroding the autonomy promise.

## Goals

### What we want to achieve

- [ ] G1: A canonical typed plan, generated by the Director, that captures phases, dependencies, acceptance criteria, and scope guards.
- [ ] G2: Per-phase Executor briefings that contain everything the Executor needs and nothing it does not, derived from the plan.
- [ ] G3: Per-phase reviews that compare the Executor's output to the criteria the Director set, with structured verdicts.
- [ ] G4: Bounded retry: a phase that fails review is re-attempted with feedback, up to a configurable retry budget, before escalation.
- [ ] G5: User-visible Director state in the TUI: which phase is active, what the brief was, what the verdict was, why.

### Success metrics

| Metric | Current | Target |
|---|---|---|
| Premature `ok` rate (phases that need rework after the loop "completed" them) | qualitative; visible | < 10% on representative specs |
| Human attention required per session-hour | full-attention | < 25% on Director-enabled runs |
| Token spend per phase | 1x baseline | 2-3x baseline (acknowledged cost) |
| Director-rejected phases per spec | n/a | a small positive number per long spec; zero is suspicious |
| Retry budget exhaustion rate | n/a | rare on representative specs; common exhaustion means the budget is wrong or the spec is unworkable |

### Non-goals

- Mid-stream sampling of the Executor (the Director only acts at phase boundaries by default).
- Auto-rewriting the spec when the plan and the spec diverge (the Director flags; the user resolves).
- Running multiple Executors in parallel (PRD 3).
- General-purpose multi-agent orchestration. The Director has one job and one tool surface.
- Replacing the Executor's internal reasoning. The Executor still does the work; the Director steers and verifies.

## Audience

| Segment | Description | Estimated volume |
|---|---|---|
| Long-spec users | Run bcc on specs that exceed a single comfortable session | Primary |
| Quality-sensitive users | Cannot tolerate silent scope drift | High value |
| Cost-sensitive users | Will keep this off for short specs; the docs make that clear | Default behavior preserves their case |

## Requirements

### Functional

#### Planning

- [ ] FR1: At run start (and on spec change), the Director produces a `Plan`: a typed object with phases, dependencies, acceptance criteria, and scope guards.
- [ ] FR2: The `Plan` is persisted to `.bcc/plan.json` and surfaced in the TUI.
- [ ] FR3: When running with `--director`, the user reviews and confirms the plan before execution begins (similar to PRD 1's confirmation step).
- [ ] FR4: When the spec changes mid-run, the Director re-plans on the next loop tick and surfaces the diff against the previous plan. If the change invalidates an in-flight phase, bcc cancels the running Executor session, asks the user to confirm the new plan, and restarts the affected phase under the new plan.

#### Briefing

- [ ] FR5: For each phase, the Director produces a `Briefing`: a structured payload the Executor adapter renders into the prompt. Contains: phase intent, acceptance criteria (verbatim from the plan), scope (files of interest, out-of-scope), distilled context summary from prior phase verdicts, retry feedback if applicable.
- [ ] FR6: The Executor receives the briefing and the spec excerpt the briefing references, not the full spec or the full journal.
- [ ] FR7: The briefing is persisted per phase to `.bcc/briefings/<phase-id>.json` for audit and `--resume`.

#### Reviewing

- [ ] FR8: After the Executor completes a phase (signal `review` per the wire protocol), the Director runs a review session that inspects the diff, the journal entry, and the original briefing. The review session is itself an agent invocation, like the Executor; bcc does not run tests or builds directly. The Director's plan declares the evidence each acceptance item requires (`diff`, `test`, `build`, `manual`); the review session gathers that evidence, including running tests or builds when required, before producing a `Verdict`: `approve | revise | escalate`, with structured feedback.
- [ ] FR9: On `revise`, bcc starts a new Executor session for the same phase with the prior briefing plus the Director's structured feedback rendered into the next briefing's `prior_feedback` field as prose. Retry budget defaults to 2; configurable per spec or overridden per phase via the plan's `retry_budget` field.
- [ ] FR10: On `escalate`, bcc pauses, surfaces the Director's reasoning to the user, and waits for input. Options: resume with hint, force-approve and advance, skip phase, abort run. This is the user's only override surface on Director verdicts; autonomous runs do not pause for user review on `approve` or `revise`.
- [ ] FR11: Verdicts are persisted to `.bcc/verdicts/<phase-id>-<attempt>.json` and rendered in the TUI.

#### Decision integration

- [ ] FR12: bcc's existing decider treats Director verdicts as authoritative for phase advancement and run completion. The Executor's own `**Result**` line is informational; an Executor `done` (whole-spec done) is interpreted as local to that session's scope and does not stop the loop while the Director has unreviewed, revising, or escalated phases. The loop terminates when the Director approves the final phase.
- [ ] FR13: The journal contains both the Executor's claim and the Director's verdict. They can disagree; the disagreement is the audit trail.

#### Schema (illustrative)

The canonical schema is finalized in the spec phase. A draft shape, for grounding the PRD:

```typescript
type Plan = {
  goal: string;                 // distilled from the spec
  success_criteria: string[];   // spec-level, not phase-level
  phases: Phase[];
};

type Phase = {
  id: string;                   // stable across re-plans
  title: string;
  intent: string;               // why this phase exists
  depends_on: string[];         // other phase ids
  parallelizable: boolean;      // hint for PRD 3
  scope_in: string[];           // files / directories
  scope_out: string[];          // explicit guards
  acceptance: AcceptanceItem[]; // checkable criteria
  retry_budget: number;         // override default 2
  executor_assignment?: ExecutorAssignment; // model, effort, optionally family. See PRD 4.
};

type AcceptanceItem = {
  id: string;
  description: string;
  evidence: "diff" | "test" | "build" | "manual"; // how the reviewer checks
};

type Briefing = {
  phase_id: string;
  attempt: number;
  spec_excerpt: string;         // the smallest relevant slice
  context_summary: string;      // from prior verdicts
  prior_feedback?: string;      // populated on retry
};

type Verdict = {
  phase_id: string;
  attempt: number;
  outcome: "approve" | "revise" | "escalate";
  acceptance_results: { id: string; passed: boolean; note: string }[];
  feedback?: VerdictFeedback;   // structured; rendered into the next Briefing's prior_feedback as prose
  reasoning: string;            // for the user, in the TUI
};

type VerdictFeedback = {
  required_changes: { acceptance_id?: string; what: string; why: string }[];
  out_of_scope: { path: string; note: string }[];
};
```

### Non-functional

- [ ] NFR1: All Director payloads are typed and round-trip through a shared `internal/director/` schema package. Same discipline as `internal/loop/agentcontract/`.
- [ ] NFR2: The Director-bcc protocol is wire-protocol-first: the Director emits structured tool calls; bcc validates and persists them. Free-form text is not part of the contract.
- [ ] NFR3: Cost reporting per Director call surfaced in the TUI; total per session reported at exit.
- [ ] NFR4: Performance: Director calls add bounded latency per phase (target: < 30s per call on typical phases). Above that, surface a slow-call warning.
- [ ] NFR5: Resume: `bcc run --resume` reconstructs Director state from `.bcc/{plan,briefings,verdicts}` without re-planning, unless the spec hash changed.
- [ ] NFR6: Determinism within reason: identical spec, identical config, similar Director responses across reruns. Some variance is inherent; no hidden randomness from bcc itself.

## Expected experience

### Story 1: long spec, hands-off

> As a user starting a 3-hour spec on Friday afternoon, I want to walk away from the laptop knowing that the Director will catch a phase the Executor mis-implemented, and either retry or pause for me, rather than blowing through and reporting `ok`.

Flow:

1. User runs `bcc run --director`.
1. Validation passes (PRD 1).
1. Director plans; user confirms.
1. Loop runs: brief, execute, review, advance.
1. On phase 5, review fails: missing test. Retry with feedback.
1. On retry, review passes. Continue.
1. User comes back; sees a clean run with one revision recorded in the journal.

### Story 2: review catches a silent scope expansion

> As a quality-sensitive user, I want the Director to flag when the Executor edits files outside the briefing's scope, so silent scope drift becomes visible.

Flow: Executor edits `internal/foo/` (in scope) and `internal/bar/` (out of scope). Director's review reads the diff, sees the out-of-scope files, returns `revise` with feedback "remove changes to internal/bar/, they are not in this phase's scope". Retry happens; phase converges.

### Story 3: escalation

> As a user, I want to be interrupted only when the Director cannot make progress, not for routine review failures.

Flow: phase 7 fails review three times (retry budget exhausted). bcc pauses, TUI shows "phase 7 escalated: Director cannot determine why the Executor keeps producing X. [R]esume with hint, [S]kip phase, [A]bort." User picks. Decision is journaled.

### Story 4: spec edited mid-run

> As an author who realized halfway through that phase 6 was wrong, I want to edit the spec and have the Director adapt, not silently keep executing the old plan.

Flow: user edits the spec mid-run. If a phase is in flight when the change lands, bcc cancels the running Executor session before re-planning. The Director re-validates and re-plans. TUI shows "spec changed: phase 6 redefined, phase 7 added. Confirm new plan?" User confirms; the loop restarts the affected phase under the new plan.

## Constraints and dependencies

- **Hard dependency** on the wire-protocol-first refactor (already in progress in MVP).
- **Soft dependency** on PRD 1: the Director can plan an unvalidated spec, at the cost of a higher revise rate. PRD 1 is recommended but not required at runtime.
- Compatible with `[executor].skip_permissions`. The Director itself is read-only on the codebase but read-write on `.bcc/` state files.
- The Director does not bypass [absolute restrictions](../../../internal/loop/agentcontract/absolute_restrictions.md) on the Executor's behalf. Verdicts cannot grant the Executor permissions the framework forbids.

## Alternatives considered

| Alternative | Pros | Cons |
|---|---|---|
| Single-agent self-review prompt (same session reviews itself) | No second session needed | Defeats the point: the same context that produced the work judges the work |
| Static rule-based review (lint diff vs. plan) | Cheap, predictable | Misses semantic gaps; can only check what we explicitly encoded |
| Always-on continuous review (mid-stream) | Catches drift earlier | High token cost; rarely actionable; deferred to a follow-up if signal warrants |
| Director and Executor as one process | Simpler wiring | Couples lifecycles; complicates parallelism (PRD 3) |
| Director writes the journal directly | Cleaner audit trail | Conflates roles; the journal is an Executor-emitted artifact today, by design |
